## What This File Does (Simple Explanation)

This file defines two small data classes that represent the results of a user's **skill assessment quiz**. When a user takes the initial 50-question assessment in the app, the answers and the generated report (as an HTML page) need to be stored somewhere — these data classes are the "containers" (blueprints) for that information.

Think of it like a form template: `AssessmentReport` is the template for the final report card, and `QuestionAnswer` is the template for each individual question-and-answer pair.

---

## Why This File Exists

The app has an **Assessment feature** where new users answer 50 questions about embedded systems. After submission, the Gemini AI generates a personalized HTML report with feedback and a 12-week learning roadmap. This file provides the data structures needed to:

1. **Store the report** in Firebase Firestore (the `AssessmentReport` class maps directly to a Firestore document)
2. **Pass question-answer pairs** to the Gemini API for evaluation (the `QuestionAnswer` class holds each Q&A pair in a compact format)

This is a **Kotlin data class** — it automatically generates `equals()`, `hashCode()`, `toString()`, and `copy()` methods. Every field has a **default value** (empty string, 0, etc.), which is required for Firebase Firestore deserialization (Firestore needs a no-argument constructor to reconstruct objects from the database).

#### Fields Explained:

| Field | Type | Default | Purpose |
|-------|------|---------|---------|
| `userId` | `String` | `""` | The Firebase Authentication UID of the user who took the assessment |
| `userName` | `String` | `""` | The display name / username of the user |
| `userEmail` | `String` | `""` | The user's email address |

> **Note**: The `reportHtml` field stores an entire HTML page as a raw string. This means every assessment report document in Firestore can be quite large (~33KB). The docs suggest this could be improved by storing structured data alongside the HTML.

### Lines 13-17: `QuestionAnswer` Data Class
```kotlin
data class QuestionAnswer(
    val n: Int,      // Question number
    val q: String,   // Question text
    val u: String    // User answer
)
```

This is a compact data class used to pass question-answer pairs to the Gemini API. The field names are deliberately **short** (`n`, `q`, `u`) to minimize the size of the JSON payload sent to the AI.

#### Fields Explained:

| Field | Type | Purpose |
|-------|------|---------|
| `n` | `Int` | The question number (1-50) — used for ordering |
| `q` | `String` | The full text of the question |
| `u` | `String` | The user's answer to the question |

> **Design Choice**: Using single-letter field names (`n`, `q`, `u`) is intentional — when this gets serialized to JSON and sent to the Gemini API as part of a prompt, shorter names save tokens (which saves money in API costs).

## Strengths

- ✅ **Simple and focused** — each class has a single responsibility
- ✅ **Firebase-compatible** — all fields have default values for Firestore deserialization
- ✅ **Compact API payload** — `QuestionAnswer` uses short field names to save API tokens

## Weaknesses / Technical Debt

- ⚠️ `totalQuestions` is hardcoded to `50` — should be dynamic if assessment length changes
- ⚠️ `reportHtml` stores the entire HTML as a string — no structured data alongside it for querying
- ⚠️ No validation — `score` could theoretically be negative or greater than `totalQuestions`
- ⚠️ Short field names (`n`, `q`, `u`) sacrifice readability for API cost savings — could use `@SerializedName` annotations instead to keep readable Kotlin names while having short JSON names
